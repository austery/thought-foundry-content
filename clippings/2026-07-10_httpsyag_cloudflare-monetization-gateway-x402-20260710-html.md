## 消失的注意力与价值交换的失灵

旧互联网的商业大厦建立在人类注意力之上。

用户浏览网页，贡献点击和时间。网站将注意力打包卖给广告商，或者引导用户进入订阅。这一注意力经济在过去三十年运转良好。

但是，AI agent 的兴起打破了这一价值交换。

AI agent
在读取网页、获取数据或调用工具时，不会看横幅广告，不在乎推荐栏，更不会为个性化推送停留。它们只发送请求，读取纯文本，随后继续执行任务。原本依靠人类注意力补贴的优质内容，在机器的高频访问下开始单向失血。内容网站陷入两难：要么彻底封杀机器访问，要么看着资产成为模型训练和生成的免费养料。

互联网需要针对机器访问的新型变现通路。

## 在边缘网络上标价：Cloudflare
的边缘飞轮

在此背景下，网络基础设施巨头 Cloudflare
开始在边缘网络部署新的经济规则。

2026 年 7 月 1 日，Cloudflare 宣布推出 Cloudflare
Monetization
Gateway。这项新服务允许客户对任何受其保护的数字资产进行按需计费，包括网页、数据集、API
和 MCP（Model Context Protocol）工具。

这不是 Cloudflare
第一次试水机器流量计费。此前，它已推出针对爬虫治理的 Pay
Per Crawl。

Pay Per Crawl 主要解决网站向 AI 爬虫单次抓取计费的问题，结算高度依赖
Stripe 等传统支付通道。新推出的 Monetization Gateway
则更进一步。它将计费对象从爬虫扩展到任意机器请求，并把计量、支付和结算从资源源站转移到
Cloudflare 全球边缘网络上。

资源方无需开发复杂的计费与验证系统。他们只需在边缘端配置规则，剩下的程序化支付和链上结算均由边缘网关处理。根据官方披露，Monetization
Gateway 启动时将支持通过 x402
协议 使用稳定币在链上完成结算。

## 价格层：从是否放行到如何报价

过去，数字资产的访问状态只有三种：免费公开、完全封禁，或者通过人工谈判签署高昂的商业合同。

这些状态均假设访问者背后是人类或企业采购团队。然而，当 AI agent
成为互联网的主要访问者，原有假设便失效了。

正在执行复杂任务的 agent 可能会临时需要查看某个网页，或者调用一次 MCP
工具。为了一次长尾调用去注册账号、绑定信用卡并购买包月套餐，交易摩擦过大。如果直接阻断，agent
就会失去完成任务的机会。

Cloudflare
Monetization Gateway
引入了第四种状态：机器可以访问资源，但访问需要携带机器可读的价格标签和付款凭证。这不是简单的付费墙，而是为非人类流量提供了一个机器可读的价格接口。

当未经验证的机器请求接入时，服务器返回 HTTP
402 Payment Required
状态码。服务器会在响应头（header）中给出报价与收款地址（参考 x402 官方指南）。agent
识别价格后，用数字钱包签署支付负载并重试请求。边缘网关验证付款后直接放行，最终在链上完成点对点资金流转。

## 从孤立的 API
实验到边缘分发基础设施

在新网关推出前，机器微支付已在小范围内完成了生产级验证。

在 《Tavily
那一分钱》 的案例中，API 检索服务商 Tavily 成功集成了 x402 协议。实验中，AI
agent 通过钱包自付 0.01 美元
USDC，顺利获取了真实搜索结果。这证明了机器自主支付的可行性。

然而，单点 API 提供商自建结算系统难以规模化推广。Cloudflare
带来的核心增量在于改变了交易发生的位置。

Cloudflare 的边缘网络覆盖全球 330
多个城市。作为网络基础设施，它本身就是大量网站、API
和工具的第一道大门。将 x402
支付协商与验证过程集成到网关层，可大幅缩短握手延迟，同时保护源站免受高频请求的干扰。

机器买家无需与卖家预先建立信任，也无需注册账号。它们只需在请求中附带付款签名，即可完成交易。微小的网络调用由此成为可在边缘随时采购的数字商品。

## 先别急着把它看成完整支付系统

到这里，容易产生一个误解：既然 agent 已经能付款，Cloudflare
又能在边缘验证付款，机器访问的商业闭环就完成了。

实际还差几层。x402 解决的是怎么报价、怎么付款、怎么结算。它不负责判断
agent 有没有被授权，不负责处理争议，也不负责区分守规矩的 agent
和恶意爬虫。整个 Agent 商业生态正快速分化为三层不同的协议：

支付与结算层：以 x402 协议 和 Stripe 的机器支付协议（MPP）为代表。这一层聚焦于程序如何在 HTTP 请求中就价格达成一致，以及资金如何快速结清。

授权与审计层：以 Google 发起的 AP2 协议和万事达卡（Mastercard）的可验证意图为代表。如 《AI 替你付款前，支付系统要先补上这条信任链》 中所分析，这一层解决法律和信任问题。x402 仅负责支付，这一层则负责核实授权主体、单次预算上限，并生成不可抵赖的加密审计凭证。

身份与防伪层：以 Visa 联合 Cloudflare 发布的 Trusted Agent Protocol 以及传统的机器人识别技术为代表。这一层用于区分合规且具备支付能力的合法 agent 与恶意的僵尸网络或爬虫，实现差异化分流与计费。

在此体系中，Cloudflare Monetization Gateway
的核心贡献，不是补完整条信任链。它做的是更窄也更关键的一步：把支付与结算层（x402）放到边缘身份识别（bot
management）旁边。网关拦截机器流量时，可以先判断来者是谁，再决定放行、阻断，还是给出价格。

这也把问题从“技术上能不能收费”，推进到下一个问题：什么资源真的值得
agent 付费。

## 价格接口不会让所有内容都变值钱

有了价格接口，不代表所有网页都能收到钱。AI agent
是否会进行单次微支付，仍然取决于资源本身是否稀缺、及时、可靠，或者能不能直接提高任务成功率。

并非所有数字资产都具备收费空间。通用内容最难收费，因为大模型大都已学习并缓存了这些常识，agent
也能在无防护站点找到替代品。对于通用网页，大厂之间通过集中授权或搜索索引进行交换更为高效。按页收费的摩擦成本极高，极易入不敷出。

相反，几类高价值数字资源将率先采用微支付模式：

实时与高动态数据：气象、路况、电商库存和金融行情等。这些数据具有极高的时间敏感度，agent 必须即时获取最新内容。

专业与结构化数据库：提供特定行业深度的垂直知识库或专利检索。这些资源能直接增强 agent 的推理结果。

高可信的 MCP 工具与执行环境：例如浏览器渲染、代码执行或特定专业计算。对 agent 来说，购买这些工具的调用权就是购买行动力本身。

这意味着，Monetization Gateway
真正推动的不是“所有内容重新收费”，而是资源市场重新分层。通用文本内容难以收费，高价值的实时数据与执行工具会最先吸引
agent 付费。

一旦资源开始分层，参与者的利益也会跟着变化。

## 价格层改变的是四方关系

当价格层嵌入互联网流量入口，变化不会只发生在内容网站一侧。资源方、API
商、agent 构建者和 Cloudflare 自己都会被拉进新的成本结构里。

内容与资源网站：短期内，Monetization Gateway 带来的是控制权和定价界面，而非暴利。网站能够据此区分人类、爬虫与合规的付费机器。但定价过高可能导致 agent 选择绕路或使用缓存；封禁过死则可能失去来自 AI 搜索引擎的推荐流量。

API 供应商：Monetization Gateway 简化了开户和签约流程，开辟了长尾分销渠道。但当 agent 能够实时比价和调用 API 时，若缺乏独家稀缺数据，API 供应商将面临更激烈的价格竞争。

Agent 构建者：过去，开发 agent 需要预先购买并配置各种第三方 API。现在，agent 获得了更广阔的长尾行动空间。但这要求构建者建立起预算控制与调用审计系统，防止 agent 陷入死循环付款或遭遇欺诈。

Cloudflare：通过在默认路径中增加计费和验证，Cloudflare 正在从安全网关向交易网关跃迁。若尝试成功，它将获得巨大的流量分发和计费控制力。然而，作为交易网关，它也将面临稳定币合规、跨国税务和反垄断监管的长期挑战。

这些影响现在还只是推演。真正的验证不在 Cloudflare
的产品公告里，而在买方是否开始把这种支付能力当成默认工具。

## 接下来要看买方，而不是看
waitlist

在机器自主支付的网络成熟前，过早宣布某种范式的终结或某种协议的获胜并无意义。卖方愿意标价，只说明它想收费。买方愿意在默认工作流里付款，才说明市场真的存在。

未来 6 到 18 个月内，以下指标比发布会更重要：

买方生态的默认支持：主流 AI SDK、MCP 客户端和 agent 编程工具（如 Claude Code 等）是否会在请求和配置中默认提供 x402 预算与审批选项。

真实收入披露：是否有资源网站或 API 商公开披露来自 Monetization Gateway 的真实收益，且该收益能稳定覆盖生产与带宽成本。

跨平台共识：该机制能否跨越 Base 和 Coinbase 联盟，在更广泛的支付网络、云巨头及边缘分发网络中达成共识。

绕过与对抗案例：恶意流量是否能通过技术手段伪装，成功绕过 Cloudflare 的边缘拦截，从而迫使防御成本螺旋上升。

基础设施巨头正尝试在边缘网络上，为非人类流量开辟自动计费通道。然而，这一通道究竟能带来持续的微支付收益，还是仅仅止步于一场协议实验，并不取决于卖方的
waitlist 长度，而取决于 agent
的成本计算模型，以及每一次携带签名付款（signed payment）的请求握手。
