在之前的视频中，我为大家演示过 **Claude Code** 的官方插件。使用这个插件，只需要一条命令，就可以将 **Claude Code** 拉入循环，从而让它全自动运行并不断迭代，直到项目达到最优。然而，当AI进行长对话时，模型会变得越来越笨拙。在长对话中，错误的尝试和冗余的信息会堆积在上下文窗口中，导致AI注意力分散。因此，在这种情况下，我们必须重置AI的“大脑”，使用一个“well-loop”循环。每次循环都能强制AI重新开始，也就是上下文重置，让AI通过读取文件（如代码、日志列表）来获取状态。这样，AI每次都是“清醒”的，并且项目的进度保存在文件系统和Git历史中，而不是在AI的对话里。这就是为什么我们可以使用这种方式，让AI全自动实现项目迭代。

这个插件虽然效果不错，尤其适合小项目的快速迭代，但当将其应用于更复杂的项目时，它在**token消耗的控制**方面就没有那么智能了，并且**缺少实时可观测性**，日志也不够完善。此外，还有用户反馈无法正常启动该插件。

### Ralph for Claude Code 登场

鉴于上述限制，本期视频将为大家演示另一款生产级别的工具：**Ralph for Claude Code**。它并非一个 **Claude Code** 插件，而是将 **Claude Code** 的能力转化为一个可执行的CLI系统。它能够让 **Claude Code** 持续自主地迭代和改进你的项目，直至完成。同时，它内置了**防死循环**和**token过度消耗**的保护机制，并具备**智能退出检测**等功能。它将AI循环技术从基础的循环升级为一个更智能的、可长期运行的开发框架。此外，它还支持**实时看板**，甚至可以直接导入现成的 **PRD** 文档或规格文档，让 **Claude Code** 按任务清单执行开发。我本人就使用 **Ralph for Claude Code**，根据产品需求文档，全自动开发了一个习惯追踪应用。

本期视频将详细演示 **Claude Code** 的使用方式、技巧以及使用后的效果。

### Ralph vs. 插件：生产级对比

在演示之前，我们先来对比一下 **Claude Code** 插件和 **Ralph for Claude Code**。它们在什么情况下各有优势？它们的区别体现在多方面，尤其是在**安全与控制**方面：
*   **Ralph for Claude Code** 支持**智能速率限制**，具有**熔断器模式**，能够防止失控和循环。它还具备**5小时AI限制检测**，能自动检测并提示等待或退出选项。此外，它拥有**智能退出检测**，能实现完成信号的语义分析。
*   相比之下，官方插件不支持速率限制，没有熔断器，也没有token限制检测。

再来看**实时仪表盘与集成能力**：
*   **Ralph** 支持**实时仪表盘**，能够实时显示状态、进度和日志。它还具有**AI驱动的语义理解**，实现响应分析。它能够直接将现有的规格文档转换为 **Ralph** 格式，并且它是独立于 **Claude Code** 的外部框架，不依赖于 **Claude Code** 的插件系统。
*   而官方插件没有项目模板，需要 **Claude Code** 的插件系统，并且完全依赖 **Claude Code**。

因此，当我们需要进行**自主运行任务**时，可以选择 **Ralph for Claude Code**。当需要**快速迭代小型任务**时，可以选择官方插件。当需要处理**现有的规格文档**，或者用于**生产环境和团队协作**时，大家可以选择 **Ralph for Claude Code**。而用于**实验或学习场景**时，可以选择官方插件。**Ralph for Claude Code** 非常强大，可以直接用于真正的项目开发和生产环境。

### 安装与配置 Ralph

使用 **Ralph for Claude Code** 非常简单。我们只需要根据官方给出的命令，直接将项目克隆到本地。然后进入项目目录，再运行 `install` 脚本，即可实现自动安装。

安装完成后，我们需要对 **Ralph for Claude Code** 进行配置，例如设置权限和请求速率等。在终端命令行，我们可以直接使用 `npm` 命令来设置 `ref_loop` 这个脚本。如果不习惯使用 `npm` 命令，我们也可以直接用 VS Code 来打开这个脚本。

在这个脚本中，可以设置**每小时的最大调用上限**，这里我设置成了100次。大家也可以根据自己的 **Claude Code** 订阅类型进行相应设置。还有一个参数是**单次调用 Claude Code 的超时时间**。在这个脚本中，还有一个重要的参数需要设置：**允许 Claude Code 可以调用哪些权限**。这里是我设置的允许 **Claude Code** 执行的权限。这几个参数都设置好之后，直接保存并关闭即可。

### PRD 文档驱动项目初始化

接下来，我们就可以根据官方给出的命令来新建项目进行测试了。在官方给出的例子中，我们可以直接提供一个已有的 **PRD** 文档给 **Ralph for Claude Code**，也可以通过手动方式，先用 `create-empty-project` 命令来创建一个空项目。

我们可以先测试一下创建空项目。直接复制这条命令，然后回到终端命令行执行即可。创建成功后，它会提示我们需要修改 `pro.md` 文件，并更新规格文档里的内容。然后就可以运行命令来启动项目开发。打开它生成的文件夹，可以看到它生成了4个 markdown 文件。这些文件是它生成的一个模板，用于给 **Claude Code** 进行参考。下面我们就可以根据自己要开发的内容，让 **Claude Code** 读取这些内容，并根据开发需求来填写。这是手动方式。

下面我们还可以使用第一种方式，即直接提供一个 **PRD** 文档。我准备的这个产品需求文档，要求开发一款移动端优先的习惯追踪应用，基于 **React Native** 和 **Expo** 构建。文档中包含了技术栈、核心功能、数据类型、UI/UX 流程以及开发优先级列表。

下面我们就可以用这条命令来初始化这个项目：`npm run import-prd -- <prd_file_path> <project_folder>`。这里我们指定了 **PRD** 文档的文件名和目标文件夹。执行后，系统会提示正在转换 **PRD** 文档，并创建了相关文件，表示 **PRD** 已导入成功。这里给出了接下来的步骤，包括审查和更改它生成的这些文件。

### 全自动开发与应用实测

如果我们感觉它生成的这些文件内容没有需要修改的地方，就可以来运行它给出的命令来启动开发。我们直接复制它给出的命令，然后切换到它为我们创建的项目路径下，粘贴并运行。这里开始启动，需要稍等片刻。

好，这里提示正在执行 **Claude Code**。下面我们可以回到项目路径来查看一下。可以看到，这里已经为我们初始化好了 **React Native** 项目。我们还可以回到项目中的 `fix_plan.md` 文件，查看它目前执行完成了哪些任务。打开后，可以看到这一部分的任务都已经完成了。像这样的话，哪怕对于再复杂的任务，我们使用 **Ralph for Claude Code** 都可以实现，不需要我们去监督，就可以运行。

在状态显示中，我们可以看到循环次数，这里已经是第三次了。再看一下它执行的任务完成情况，可以看到这些任务基本都快完成了，还有个别未完成的。这里我们再稍等一下。循环次数已经到了第四次。

在等待了大约20分钟左右，这里完成了开发。下面我们在模拟器中查看一下它为我们开发的这个习惯追踪应用的效果。底部是这三个标签页。点击“添加”按钮，可以输入内容，选择图标和颜色。还可以开启提醒并设置时间。设置好后点击保存。

这里提示这个习惯追踪已经添加成功。我们还可以对习惯进行编辑修改，也可以点击删除。点击“统计”，就可以看到详细的统计内容，包括刚才添加的习惯，还可以按周、按月、按年进行分类。在“设置”里，可以进行主题颜色修改、开启通知、触觉反馈，查看今天已完成的习惯，还可以导出数据。

这样，我们就使用 **Ralph for Claude Code** 为我们开发了一个习惯追踪的 App。通过测试可以发现，**Ralph for Claude Code** 比之前的插件效果要更好。最关键的是它支持我们直接用文档来初始化项目，而插件只支持简单的任务描述。对于复杂项目，大家就可以使用 **Ralph for Claude Code** 进行全自动开发。
