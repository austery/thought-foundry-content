### 评测的两个认知误区

很多人在开发AI系统时常常感到孤立无援，但我今天的核心主张是：人们对**评测**（Evals）的理解往往是错误的。目前在行业内，关于评测存在两个极端的认知阵营：

第一个是**客观指标派**（Objective Metrics Camp）。这一派倾向于“刷榜（Benchmark Maxing）”，他们看着仪表盘上的分数，就天真地认为具有相似高分的模型（例如某些号称媲美前沿模型的测试版本）在能力上是等同的。但一旦将这些模型置于真实的实际测试中，你就会发现它们并不一样，表面的高分往往经不起现实世界证据的检验。

第二个是**纯凭感觉派**（Taste Camp）。这一派走向了另一个极端，他们认为一切都只关乎“感觉（Vibes）”，甚至会像评价歌剧一样去拟人化地评价大语言模型（例如说“我喜欢和她聊天”）。

事实上，真相介于两者之间。评测既不是衡量一切的终极标准，也并非毫无用处。我们需要掌握正确的方法去使用它们。在深入构建自定义评测之前，我们首先需要建立解读评测的直觉。

### 解读评测的三大启发式原则

要想有效利用评测，我们可以将其分为三个阶段：借用他人的评测、利用评测改进自己的智能体（Agent），以及针对特定场景构建自定义评测。为了帮助大家建立正确的判断框架，我总结了三个启发式原则：

*   **对分数保持警惕**：当模型厂商发布带有高分数据的应用时，不要轻易相信。很多AI研究人员和工程师已经习惯性地忽略这些数字，因为它们通常只是一种近似值。
*   **避免盲目追新**：保持对前沿技术的关注，但不要总是做**最早的采用者**（Earliest Adopter）。观察 Epoch Index 就会发现，前沿模型每隔几个月甚至更快就会更替一次。更好的策略是让新模型先沉淀几周，如果它经历了时间的考验，再去进行模型切换，而不是疲于奔命地追赶测试。
*   **寻找最新且精准的评测**：许多标准化的评测实际上已经过时了。OpenAI 曾在博客中明确指出，类似 SWE-bench 这样的旧指标已经无法衡量前沿模型的编码能力。因为它们还在测试如何解答斐波那契数列或进行矩阵乘法，而这根本无法代表真实的软件工程问题。你需要的是具备极高分辨力的新型评测体系。

### 构建真实的工程环境评测

当我们利用评测来改进智能体时，必须将其视为一个工程与哲学的双重难题。在工程层面上，环境隔离极其困难；在哲学层面上，你需要构建一种能够近似反映真实世界无限问题空间的测试框架。

在 **Cline**（一家开源的编程智能体公司），我们发现过去市面上的单轮评测（Single-turn Eval）过于初级，就像在问大语言模型“猫有几个脚趾”并期待一个二元答案。但当你评估一个复杂的智能体时，任务往往是“帮我修复这个未知 MCP Server 的错误”。智能体会去阅读文件、搜索文档、配置环境、运行 Python 脚本并执行测试，这一连串动作可能需要长达数十分钟。单纯看最终结果，很难衡量它是否在此过程中破坏了其他代码。

为了解决这个问题，类似 **Terminal Bench** 这样的工具应运而生。它包含了89个真实的编程任务（如竞态条件、数据库问题、基础设施问题）。为了安全有效地运行它，你必须为每一个测试任务提供一个完全隔离的虚拟机环境，在其中加载整个代码库并运行你的智能体（如 Cline、Claude Code 或 Codex）。为了不被最慢的任务拖垮进度，你可以利用像 **Harbor** 这样的开源软件配置标准化的底层基础设施，并借由 **Modal** 这样的算力平台，将这89个真实环境隔离的复杂任务并行跑完。

### 基于评测的渐进式攀登与优化系统

建立起并发的评估流水线后，真正的挑战是如何提升智能体的分数（**Hill Climbing**）。通过分析每次测试产生的大量大语言模型调用记录（Trace），你需要定位失败的具体原因，并精准规划优化策略。本质上，评估过程是在交叉验证以下三个维度：

1.  **底座模型**：模型自身的推理水平是否足够强大。
2.  **工程线束（Harness）**：你构建的代理框架是否能充分释放特定模型家族的潜力（比如 Claude Code 就将 Anthropic 的模型能力发挥到了极致）。
3.  **问题的真实性**：你是否在解决对现实有真正意义的复杂难题。

针对这些发现，我们可以将优化工作划分为三个层次：
*   **第一重优化（Obvious Flaw）**：修复明显的系统级物理故障。例如运行环境配置错误、API 频率限制（Rate Limiting），或者是智能体的重试机制代码本身存在导致框架崩溃的 Bug。
*   **第二重优化（Nuance Improvements）**：这是决定系统上限的核心区域。你需要针对特定的模型系列进行深入的**提示词工程**（Prompt Engineering）。例如，给模型强制设定过长的思考空间有时反而会导致它陷入死循环（比如输出几千个毫无意义的自我复读 Token）。通过调整和剪裁提示词尺寸，挖掘各个模型的独特能力优势。
*   **第三重优化（The Danger Zone）**：也就是系统性的**过拟合（Overfitting）**。纯粹为了拿高分而作弊，虽然能让你发一条充满噱头的推文，但毫无真实的工程意义，必须坚决避免。

最终，对待任何复杂的系统研发任务，正确的方法论应该是：找到一个能反映真实情况的评测基准，构建严密的自动化执行管道，然后不断攀登与优化。你需要让系统不仅在客观分数上不断进步，同时在实际使用体验（Vibe Check）上也能给用户提供真正流畅可靠的反馈。
