我们先来看动态工作流（dynamic workflows）
动态工作流本质上是一段 JavaScript 脚本
用来大规模编排子Agent
用户描述一个任务之后
Claude 会为这个任务编写脚本
然后由一个运行时在后台执行
与此同时
用户当前的会话仍然可以保持响应
不会因为子agent在后台工作而卡住
它的核心变化在于
任务计划被转移到了代码里
而不是继续塞在 Claude 的上下文窗口中
中间结果也会保存在脚本变量里
因此
Claude 的上下文里只保留最终答案
这也是动态工作流和skill的关键区别
用户可以通过两种方式来启动一个工作流
第一种是在提示词的任意位置包含 workflow 这个词
第二种是开启一个名为 ultracode 的设置
Ultracode 会把 xhigh 级别的推理强度与自动 工作流编排结合起来
Claude Code 还内置了 deep-research
它本身就是一个预置 workflow
当一个工作流启动后
Claude 会根据用户的提示词动态规划任务
把任务拆解成多个子任务
然后把工作分发给并行运行的子Agent
这些子agent会从彼此独立的角度处理问题
另外一些子agent则会尝试反驳这些发现
整个运行过程会不断迭代
直到答案逐渐收敛
而且结果在被纳入最终输出之前
还会经过检查
同时
工作流运行时会施加明确的硬性限制
最多允许 16 个子agent并发运行
单次运行的子agent总数上限为 1000 个
工作流脚本本身不能访问文件系统或 shell
只有 子agent可以读取、写入文件并运行命令
随着运行推进
工作流的进度会被保存下来
如果任务中途被打断
它可以在同一个会话中恢复
已经完成的子agent 会在恢复时返回缓存结果
由于协调过程发生在对话之外
所以任务计划也就更容易保持稳定
不会在会话上下文中被不断得稀释或者带偏
Anthropic 重点展示了一个大型案例
Bun框架的作者贾里德·萨姆纳（Jarred Sumner） 使用动态工作流对 Bun 进行移植
想把 Bun 从 Zig语言迁移到 Rust语言
其中一个工作流负责为每个 struct 字段
映射正确的 Rust lifetime
下一个工作流则把每个Rust文件写成行为一致的移植版本
几百个子agent并行工作
并且每个文件都有两个审查员
随后
一个修复循环会持续驱动构建和测试套件
直到全部清理干净
最终
动态工作流总共生成了大约 75 万行 Rust 代码
通过了现有测试套件的 99.8%，
并且从第一次提交到最终合并
仅用了 11 天
不过

仅面向 Max、Team 和 Enterprise 计划开放
其中在 Max 和 Team 计划中它是默认开启的
但是在 Enterprise 计划中

它既可以在 CLI、Desktop 和 VS Code 插件中运行
也可以在 Claude API、Amazon Bedrock、Vertex AI 和 Microsoft Foundry 上运行
