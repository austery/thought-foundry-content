PostgreSQL是世界上最安全的系统
你没听错
不只是最安全的“数据库系统”，是最安全的“系统”
没有之一
简单总结就是：
它有着一套360度全环绕的安全机制
可以做到彻底无死角
把安全落实到每一行、每一列，乃至到每一个单元格里
不过在深入探讨它的各种安全功能之前
有一个更重要的问题我觉得得先聊聊：
那就是计算机行业里
对于数据库安全的重视程度的严重缺失
著名程序员Atomic Energy说过
“系统安全的本质就是数据读写的安全”
因为古今中外，我们可以把所有的安全事故分为三种：
第一种就是各种机密信息的泄露（READ）
这些事故数不胜数，甚至可以说养活了半个暗网
而第二种，就是系统遭遇到不可逆的破坏（WRITE）
其他程序被破坏了还可以重启
而数据库作为系统里面的single point of truth
数据被修改或者是删除了，就无法挽回了
一个经典案例就是2017年的GitLab事故
在一连串的教科书式的
搞笑程度不亚于去年那几次大型互联网瘫痪事故的
究极草台班子操作之后
GitLab的程序员
删掉了托管在自己家PostgreSQL里的300GB的用户数据
而且无法恢复
而最后一种安全事故
就是系统受到干预而无法正常使用的情况了（DOWN）
比如勒索木马和DDoS袭击
但这种更多的是网络架构问题
可以看到这里有2/3都是跟数据读写相关的
所以本质上就是“谁可以读取或者写入什么数据”的问题
换个角度来说，也就是“身份验证和权限管理”的问题
然而在业界却有一个很诡异的现象
就是身份和权限越在外层就越严防死守
而越进入到数据库层面，反而就变得很随便了
比如在前端，我们就见识了浏览器这几十年的发展
从一个四处漏风的筛子，进化成现在固若金汤的沙盒
而在前后端的通讯上
我们也见证了几代SSL和TLS协议的升级
也经历了HTTPS从小众变成默认的变迁
后端的安全架构更是百花齐放了
以前随便暴露IP、暴露SSH端口的事情
现在基本上也不怎么发生了
然而当数据流到终点，到达数据库的时候
画风忽然就变潦草了
比如我见过很多大型的项目
都是直接在后端用admin账号直连数据库
几乎什么防御都没有
如果硬说要有
就只是用一些加密手段把admin账号和密码传入到后端的代码里
我也理解这样做的好处
首先是可以减少数据库账号的数量，容易管理
二就是应用层的代码写起来就比较简单
还有在性能上也有优势，这个后面（章节）会聊到
但我认为很多人这样做
更多的原因是存在一个侥幸心理
很多人会觉得后端是部署在自己家里的
家里已经做好了安全防范了
那后端连数据库就相当于客厅连卧室
就不用搞那么复杂了吧？
这种侥幸心理的结果就是：
让闯进家里的人，不管是故意的还是不小心的
都直接手握了admin账号这个核弹
毁天灭地就是一念之间
我之所以想要唤起大家对数据库安全的重视
正是因为只有数据库可以救自己
著名程序员Atomic Energy说过
“安全是有覆盖范围的，超出了范围一律要默认成为不安全”
我们就拿“身份”这个数据来说
从浏览器沙盒里出来，中间有HTTPS加密
来到后端了，我们还要重新验证这个身份
为什么呢？
因为它的每次转手，都脱离了前者的安全框架的覆盖范围
那对于后者来说，数据到手的时候就是不可信的了
同理，当数据库接收到某个外部请求的时候
甭管它是来自于哪里
只要是在数据库的安全范围之外，我们就应该一视同仁
对它们做全方位的身份和权限的验证
这个操作本质上和其他应用层没有多大区别
就拿我们的主角PostgreSQL来说
它的安全机制是围绕着role（角色）这个概念来设计的
不管是要打造沙盒环境
还是要做身份和权限的认证
我们都得基于自己创建这个角色
现在你可以简单地把它理解为数据库的登录账号
至于具体怎么设计它，我们后面（章节）会再说

和MySQL这种残次品不同的是
PostgreSQL是一个正常的数据库
所以它有database、schema和table的三层结构
我们都知道一个transaction是不可以跨database执行的
但是你可以在同一个database里面
跨schema去查询不同的table
schema之间是没有隔离的
那除了帮强迫症去整理归类各种table
达成一种namespace的效果之外
schema还有什么存在价值吗？
那当然是有的
因为在PostgreSQL里有一个叫做search_path的功能
它可以给不同的角色制定不同schema的可见度
比如你可以让不同部门的系统只能够看到自己部门的schema
然后给一些跨部门运行的系统看到好几个部门的schema
如果你的系统架构没有那么复杂
也可以像我这样，只定义三个简单的schema：
那些用户可以直接查询到的table，都放到public schema里
而储存的账号、密码、session之类的机密信息的table
则放到private schema里
用户角色的search_path里面没有private schema
就没有办法查询到这些table
只能够通过固定的trigger或者function间接获取
我们就可以控制用户对机密信息的获取能力
而那些和用户完全无关的table
比如说在后台自动运行的异步服务
就可以放到只有它们自己能够看到的worker schema里
确保不会受到人工query的干预或者副作用的影响
所以在这里
我基本上就是把schema当做是沙盒来用了
把不同安全等级的table互相隔离开来
这个就是PostgreSQL提供的第一层防御机制
那么schema权限通过后，我们正式进入到这个table里
但PostgreSQL的table不像一个Excel表那样
一打开就可以一览无遗
在默认状态下，一个角色没有权限读写任何的column
你得手动授予权限
而且CRUD是要分开授权的
比如在这个查询里
我们要找用户的ID和名字
那就得用GRANT命令，给发起查询的这个角色
授予 ID 和 name 这两个column的SELECT权限
当然我也见过不少人偷懒
直接把所有table的所有column的所有CRUD权限
都一次性地授予了所有的新建角色
这样做确实省事很多
但如果你的目的是省事的话
我建议直接“DROP TABLE”，一步到位
所以在column层面，我的推荐就是：
“只给用户最低程度的使用权限”
比如这个用户的会员等级，用户自己肯定是只能看不能改的
那我们就只给他/她SELECT的权限
又比如我们同步了用户的微信账号
拿到了捆绑的微信openID、unionID这些参数
这些东西虽然是属于用户的，但他/她自己用不上
只有后台程序调用API的时候需要
那么我们就连它们的SELECT权限都不应该给这个用户了
这个问题其实是需要得到重视的
前些天，我看到一个安全机构的研究报告
说是有96%的企业权限设置是“空置”的
也就是说这个权限被创建了
但是一直没有人用，也没有被关掉
报告指出：当AI开始接管操作的时候
这些可以被任意使用的权限范围将会是超大的安全隐患
别的不说，我已经能够想象出这个画面了
所以在授权问题上
我建议大家遵循“只做加法、不做减法”的原则
就能避免不经意间把不必要的权限泄露出去
column层面的权限验证通过后，我们就会来到最后的row层面
这里也是PostgreSQL最无敌的地方
因为它有一套独一无二的，图灵完备的动态规则系统：
Row Level Security（RLS）
前面介绍的 search_path 和 GRANT 都是静态的匹配逻辑
你执行一行命令
就可以把一个 schema 或者 column 的权限授予一个角色
这个角色就能一直拥有这个权限
直到你再执行一个命令剥夺它
而RLS是完全动态的
它会在每一个transition发起的时候
实时地验证一个角色的CRUD权限
比如这里有一个订单table
我们想要让买家只能够读写自己的订单
店家更是只能读不能写
我们就可以在用户发起请求时
通过PostgreSQL的运行时参数功能 pg_settings
把用户的ID注入到transaction里面
然后在这个查询到达RLS验证时
把用户ID取出来，直接对比当前这个row的买家和店家ID
在UPDATE权限的policy里就检查他是不是买家
在SELECT权限的policy里就检查他是不是买家或者店家
这样就算你是玉皇大帝，来到这最后一步
你的ID对应不上这一个row的买家或者店家ID
你也没有权限读取这个row的数据
所以RLS就是整个PostgreSQL安全机制的最后一道锁
也是最强大的
因为RLS执行的验证逻辑可以是任何形式的SQL语句或function
而在function里你也可以实时调用任何数据用来配合验证
比如说我们要追加一个权限：
允许这个店铺的员工读取本店的所有订单
那么在RLS验证的时候
我们就可以先查询储存员工数据的table
获取这个用户正在任职的店铺ID
再回来订单table去匹配这个订单的店铺ID
而正因为这个验证是实时的
在员工被解雇时
员工信息的table更新的那一瞬间
他就失去了所有订单的读取权限
完全的零延迟
如果你没有性能上的担忧，没有TPS的瓶颈
那你还可以再极端一点：
比如把所有系统层面和业务层面的验证逻辑
全部都塞进那一个function里，给RLS做实时的验证

比如你（又）想搞出最极端的做法
就可以回到最顶层的角色（role）
可以给每一个用户都创建一个单独的角色
这样每个用户都能够从schema层面就开始单独的分配权限
当然我不建议这么做
因为数据库有很多性能节点是跟角色挂钩的
比如connection pool里面的connection
只能在同一个角色之间重用
你有很多个角色的话，每个角色都要经历冷启动
还得抛弃不少只能够在（同一个）角色之间共用的缓存
这几乎能够把【让PostgreSQL再次伟大#01】里

这也是视频开头说到的
很多项目喜欢一个admin角色用到底的最后一个原因：
可以避免角色切换导致的性能损失
但作为一个很喜欢折中的中国人
我的建议是：
既不要给每个用户都创建角色，也不要只用一个admin
而是找一个兼顾安全和性能的平衡点
比如我就设计了一个“三权分立”的角色结构：
第一个connector，只给了发起数据库连接的权限
后端程序用这个角色连上了connection pool
就可以避免稀释连接池，或者切换角色的冷启动
当用户发起具体的查询请求时
就像前面提到的，后端会把用户ID注入到运行参数里
同时通过 SET ROLE（命令）切换到第二个角色visitor
而这个角色
就是数据库里面所有的search_path、GRANT和RLS设置里
所面对的那个角色
这样子，后端程序就只使用第一个角色
而数据库的所有的验证功能都只和第二个角色打交道
两边就可以处于相对独立的状态
而最后一个角色，就是传统意义上的admin了
但它只负责在初始阶段创建一切
不在日常的运行中使用
总的来说
这个设计可以在很大程度上把不同的权限范围隔离开
避免不必要的角色权限被滥用或者被泄露
同时避免了角色切换对性能的破坏
